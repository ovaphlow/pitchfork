# 药房模块标准化设计（精简版）

## 1. 模块概述

药房模块作为独立 Gradle 模块（`:pharmacy`），是连接医疗医嘱与库存实物的桥梁。核心职责包括：处方接收与审方、药品调配与核对、发药确认、退药处理，以及科室/护理站申领管理。模块遵循单向依赖原则：`healthcare` → `pharmacy` → `inventory`。通过配置项与角色权限，同一套表结构可同时适配严格 HIS 与轻量养老院场景。

## 2. 业务流程

### 2.1 整体流程
```
医嘱开立 → 药房接收 → 审方（可配置） → 调配 → 核对（可配置）
→ 发药确认 → 库存出库 → 发药完成
         ↓
      退药申请 → 审核 → 库存入库 → 退药完成
```
### 2.2 申领流程
```
科室/护理站发起申领 → 药房审批 → 调配 → 发放 → 库存调拨出库
```
### 2.3 场景裁剪

| 步骤 | HIS 医院 | 养老院 |
|------|----------|--------|
| 审方 | 必须药师审方 | 可自动通过，或由护士长审方 |
| 调配核对 | 双人核对 | 单人操作 |
| 申领管理 | 病区向中心药房请领 | 护理站向药房/总库申领 |
| 与医嘱关联 | 严格关联 `order_item_id` | 可简化或批量关联 |

## 3. 数据库结构（DDL）

所有表位于 `healthcare` schema（可独立 schema），遵循统一规范：表名复数、主键 `VARCHAR(32)`、状态/类型 `VARCHAR` + `CHECK`、扩展 `metadata` JSONB。

```sql
-- =====================================================
-- 药房模块 - 核心表结构
-- 发药、退药、申领
-- =====================================================

CREATE SCHEMA IF NOT EXISTS pharmacy;
SET search_path TO pharmacy;

-- 发药单主表
CREATE TABLE pharmacy_dispenses (
    id              VARCHAR(32) PRIMARY KEY,
    dispense_no     VARCHAR NOT NULL UNIQUE,           -- 发药单号
    patient_id      VARCHAR(32) NOT NULL,              -- 患者/老人ID（外键由 healthcare 模块管理）
    encounter_id    VARCHAR(32),                       -- 就诊/入住ID
    dispense_type   VARCHAR NOT NULL CHECK (dispense_type IN (
                        'OUTPATIENT',       -- 门诊发药
                        'INPATIENT',        -- 住院发药
                        'WARD_BATCH',       -- 病区统领
                        'ELDERLY_ROUTINE'   -- 养老院日常发药
                    )),
    status          VARCHAR DEFAULT 'PENDING' CHECK (status IN (
                        'PENDING',          -- 待处理
                        'REVIEWED',         -- 已审方
                        'DISPENSING',       -- 调配中
                        'DISPENSED',        -- 已发药
                        'CANCELLED'
                    )),
    pharmacist      VARCHAR,                           -- 调配药师/发药人
    reviewer        VARCHAR,                           -- 审方药师/核对人
    metadata        JSONB,                             -- 扩展：病区、特殊标记等
    created_at      TIMESTAMPTZ DEFAULT now(),
    dispensed_at    TIMESTAMPTZ
);

-- 发药明细
CREATE TABLE pharmacy_dispense_items (
    id                          VARCHAR(32) PRIMARY KEY,
    dispense_id                 VARCHAR(32) NOT NULL REFERENCES pharmacy_dispenses(id),
    order_item_id               VARCHAR(32),           -- 医嘱明细ID（弱关联）
    order_execution_id          VARCHAR(32),           -- 医嘱执行记录ID
    material_id                 VARCHAR(32) NOT NULL,  -- 药品ID（弱关联）
    lot_id                      VARCHAR(32),           -- 批次ID
    prescribed_quantity         NUMERIC(15,4),         -- 处方数量（包装单位）
    dispensed_quantity          NUMERIC(15,4),         -- 实发数量
    unit                        VARCHAR DEFAULT 'PACKAGE' CHECK (unit IN ('PACKAGE','SPLIT')),
    split_quantity              NUMERIC(15,4),         -- 拆零数量
    stock_operation_detail_id   VARCHAR(32),           -- 关联库存出库明细ID（弱关联）
    unit_cost                   NUMERIC(15,4),
    total_cost                  NUMERIC(18,4),
    metadata                    JSONB
);

-- 退药单主表
CREATE TABLE pharmacy_returns (
    id                  VARCHAR(32) PRIMARY KEY,
    return_no           VARCHAR NOT NULL UNIQUE,
    original_dispense_id VARCHAR(32) NOT NULL REFERENCES pharmacy_dispenses(id),
    patient_id          VARCHAR(32) NOT NULL,
    return_reason       VARCHAR,
    status              VARCHAR DEFAULT 'PENDING' CHECK (status IN ('PENDING','CONFIRMED','CANCELLED')),
    operator            VARCHAR,                        -- 操作人
    metadata            JSONB,
    created_at          TIMESTAMPTZ DEFAULT now(),
    confirmed_at        TIMESTAMPTZ
);

-- 退药明细
CREATE TABLE pharmacy_return_items (
    id                          VARCHAR(32) PRIMARY KEY,
    return_id                   VARCHAR(32) NOT NULL REFERENCES pharmacy_returns(id),
    dispense_item_id            VARCHAR(32) NOT NULL REFERENCES pharmacy_dispense_items(id),
    quantity                    NUMERIC(15,4) NOT NULL,   -- 退回数量（包装单位）
    stock_operation_detail_id   VARCHAR(32),              -- 关联库存入库明细ID
    unit_cost                   NUMERIC(15,4),
    total_cost                  NUMERIC(18,4),
    metadata                    JSONB
);

-- 药品申领单主表（科室/护理站向药房请领）
CREATE TABLE pharmacy_requisitions (
    id              VARCHAR(32) PRIMARY KEY,
    requisition_no  VARCHAR NOT NULL UNIQUE,
    warehouse       VARCHAR NOT NULL,                    -- 请领的药房仓库名称
    requester       VARCHAR,                             -- 申领人
    department      VARCHAR,                             -- 申领科室/护理站
    status          VARCHAR DEFAULT 'DRAFT' CHECK (status IN ('DRAFT','APPROVED','DISPENSED','CANCELLED')),
    metadata        JSONB,
    created_at      TIMESTAMPTZ DEFAULT now(),
    dispensed_at    TIMESTAMPTZ
);

-- 申领明细
CREATE TABLE pharmacy_requisition_items (
    id                          VARCHAR(32) PRIMARY KEY,
    requisition_id              VARCHAR(32) NOT NULL REFERENCES pharmacy_requisitions(id),
    material_id                 VARCHAR(32) NOT NULL,
    requested_quantity          NUMERIC(15,4) NOT NULL,   -- 申领数量
    approved_quantity           NUMERIC(15,4),            -- 批准数量
    dispensed_quantity          NUMERIC(15,4),            -- 实发数量
    stock_operation_detail_id   VARCHAR(32),              -- 关联库存调拨出库明细ID
    metadata                    JSONB
);

-- 索引建议
CREATE INDEX idx_dispense_patient ON pharmacy_dispenses(patient_id);
CREATE INDEX idx_dispense_items_dispense ON pharmacy_dispense_items(dispense_id);
CREATE INDEX idx_return_patient ON pharmacy_returns(patient_id);
CREATE INDEX idx_requisition_department ON pharmacy_requisitions(department);
```

## 4. 与库存模块的集成

- **发药**：确认发药时，药房调用库存出库接口。传入 `warehouse`、`material_id`、`lot_id`、`quantity`、`unit_cost` 等参数，库存模块生成 `stock_operations`（`operation_type='DISPENSE'` 或 `'OUTBOUND'`）及明细，返回 `stock_operation_detail_id`，由药房模块存入 `pharmacy_dispense_items.stock_operation_detail_id`。
- **退药**：确认退药时，调用库存入库接口，生成 `stock_operations`（`operation_type='RETURN'` 或 `'INBOUND'`），同样回写明细 ID。
- **申领**：确认申领发放时，调用库存调拨接口，从药房仓库调拨至科室/护理站仓库，生成 `stock_operations`（`operation_type='TRANSFER'` 或 `'OUTBOUND'`），明细回写。

所有库存操作均由库存模块独立完成，药房模块只保存关联 ID，不直接操作库存表。

## 5. 与医疗模块的集成

- 药房模块通过 `patient_id`、`encounter_id` 获取患者信息，通过 `order_item_id`、`order_execution_id` 关联医嘱（弱引用）。这些外键不强制数据库约束，而是通过 Service 接口访问 `healthcare` 模块。
- 发药完成后，药房可通知 `healthcare` 模块更新医嘱执行状态（如将对应 `order_execution` 标记为已完成），或由 `healthcare` 模块自行轮询/事件触发。

## 6. HIS 与养老场景差异化配置

| 配置点 | HIS 模式 | 养老模式 |
|--------|----------|----------|
| 审方要求 | `status` 必须经过 `REVIEWED`，需要 `reviewer` | 可配置跳过审方，`status` 直接从 `PENDING` 到 `DISPENSING` |
| 双人核对 | `reviewer` 与 `pharmacist` 不能为同一人 | 允许同一人 |
| 发药类型 | `OUTPATIENT`、`INPATIENT`、`WARD_BATCH` | 增加 `ELDERLY_ROUTINE` |
| 与医嘱关联 | 严格关联 `order_item_id` | 可批量发药，医嘱关联可选 |
| 申领部门 | 病区、科室 | 护理站、楼层 |
| 费用联动 | 发药自动计费，退药自动退费 | 通常不计费或月结 |

这些差异通过系统配置参数或机构类型在应用层控制，数据库结构保持统一。

## 7. 毒麻药品与盘点处理

- **毒麻药品**：不再单独建表，可在发药/退药明细的 `metadata` 中增加 `"narcotic": true` 标记，并在业务层强制要求双人核对字段填写。如有专项统计需求，可通过查询发药记录加上过滤条件实现。
- **药房盘点**：直接复用库存模块的 `stock_checks` 表，指定 `warehouse` 为药房仓库即可。无需额外建表，盘点差异由库存模块统一处理。

---

此设计在保证药房业务完整性的前提下，以 4 张核心表（发药、发药明细、退药、退药明细、申领、申领明细，实际6张）覆盖了发药、退药、申领三大流程，结构极简，兼容性强，可直接作为通用 HIS 与养老平台的药房标准化方案。
